# ФАЗА 1 — БИЗНЕС И ПРОДУКТОВАЯ ЖИЗНЕСПОСОБНОСТЬ

## 1.1 Покрытие бизнес-процессов

Система покрывает следующие финансовые и операционные процессы:

| Модуль | Процесс | Зрелость |
|--------|---------|----------|
| **Склад (Inventory)** | Учёт ТМЗ металлопроката: трубы, профили, листы, балки. Два склада (основной, облачный). Мин. уровни запасов. | Базовый |
| **Закуп (Procurement)** | Закупки у поставщиков с расчётом landed cost: логистика, таможня, импортный НДС. Конвертация UZS→USD. | Средний |
| **Касса (Sales)** | Оформление продаж, смешанные методы оплаты (нал/безнал/карта/долг/микс), чеки, счета-фактуры. | Средний |
| **Workflow** | Предзаказы: draft → confirmed → sent_to_cash → completed/cancelled. Конвертация в заказы. | Базовый |
| **CRM** | Клиентская база (физ./юр. лица), кредитные лимиты, история покупок, долги. | Базовый |
| **Баланс** | Управленческий баланс: Активы = Пассивы. Стоимость ТМЗ, касса, дебиторка, кредиторка. | Средний |
| **PnL** | Отчёт о прибылях и убытках: выручка, себестоимость, валовая прибыль, OPEX по категориям. | Средний |
| **Основные средства** | Учёт ОС с амортизацией по 11 категориям (здания 5%, компьютеры 20%, земля 0%). | Базовый |
| **Зарплата (Payroll)** | Расчёт зарплаты, KPI с комиссией от прибыли, авансы. | Минимальный |
| **НДС** | Входящий/исходящий НДС, расчёт обязательств (12% ставка Узбекистана). | Базовый |
| **Журнал** | Аудит-лог операций сотрудников (append-only, immutable). | Базовый |

## 1.2 Корректность доменного моделирования

### Сильные стороны:
- **Двухвалютная система** грамотно реализована — USD как базовая валюта для бухгалтерии, UZS для кассы и поставщиков. Курс фиксируется на момент операции (`exchangeRate` в каждом документе).
- **Snapshot-подход к финансам** — `vatRateSnapshot`, `costAtSale`, `priceAtSale` фиксируют условия на момент сделки, что корректно для аудита.
- **Landed cost** в закупках — отдельный расчёт себестоимости с учётом логистики, таможни, НДС.
- **Workflow-воронка** — логичный жизненный цикл заказа от заявки до оплаты.
- **Versionable** — базовый интерфейс для оптимистичной конкурентности (правда, не везде применяется).

### Критические слабости:
- **Отсутствие двойной записи (Double-Entry Bookkeeping).** Это ФУНДАМЕНТАЛЬНЫЙ дефект для ERP-системы. Нет ни плана счетов, ни проводок, ни дебет/кредит журнала. Система использует **однократную запись + денормализованные счётчики** (`totalDebt`, `totalPurchases` на клиенте). Это делает НЕВОЗМОЖНЫМ:
  - прохождение профессионального аудита
  - сертификацию по стандартам бухгалтерского учёта
  - корректную трансформацию отчётности

- **`retainedEarnings` — балансирующая затычка.** Нераспределённая прибыль вычисляется как `totalAssets - equity - fixedAssetsFund - vatLiability - accountsPayable - fixedAssetsPayable`. Это маскирует любые ошибки — баланс ВСЕГДА сходится математически, но не фактически.

- **Привязка клиент↔заказ по имени.** `useDebtRecalculation` сопоставляет клиентов с заказами по `clientId`, имени и `companyName` через string-matching. Переименование клиента = потеря связей с долгами.

- **Нет периодов (Accounting Periods).** Нет понятия закрытия месяца/квартала/года. Все данные — один непрерывный поток. Невозможно заблокировать редактирование прошлых периодов.

## 1.3 Логичность учётных потоков

| Поток | Корректность | Проблема |
|-------|-------------|----------|
| Продажа → инвентарь | ✅ Атомарно (salesAtomicService) | — |
| Продажа → долг клиента | ✅ Атомарно | — |
| Закупка → инвентарь | ❌ Не атомарно | purchaseService.add() НЕ обновляет остатки |
| Удаление заказа → инвентарь | ❌ Не реверсируется | Товар не возвращается на склад |
| Оплата долга → клиент | ✅ Атомарно (transactionService) | — |
| Удаление транзакции → долг | ⚠️ TOCTOU уязвимость | Чтение вне транзакции |
| Возврат товара | ⚠️ Частично | Нет чёткого процесса реверса |

## 1.4 Защита от финансовых несоответствий

**Слабая.** Основные риски:

1. **Инвентарь и финансы могут рассинхронизироваться** при сбое между записями (закупка без обновления склада, удаление заказа без возврата товара).
2. **Баланс не имеет независимой верификации** — единственный механизм проверки — это пересчёт ВСЕх данных в браузере при каждом изменении.
3. **Нет контроля закрытия периодов** — любую прошлую операцию можно изменить, что нарушает аудиторский след.

## 1.5 Устойчивость бизнес-модели

Продукт позиционирован как **нишевая ERP для металлоторговли в Узбекистане**. Это узкий, но реальный рынок. Однако:

- **Нет SaaS-инфраструктуры** — приложение деплоится на GitHub Pages, данные в единственном проекте Firebase. Нет мультитенантности, нет биллинга, нет onboarding-потока.
- **Нет API для интеграций** — невозможно подключить банковские системы, 1С, налоговые API Узбекистана.
- **Нет мобильного приложения** — только адаптивный веб.

## 1.6 Enterprise-позиционирование

**НЕ ВОЗМОЖНО в текущем состоянии.** Для enterprise-клиентов критически необходимы:
- Двойная запись ✘
- Мультитенантность ✘
- RBAC с аудитом ✘ (частичный)
- SLA и мониторинг ✘
- API интеграции ✘
- Сертификация соответствия ✘

# ФАЗА 2 — ОЦЕНКА ENTERPRISE-АРХИТЕКТУРЫ

## 2.1 Тип архитектуры

**Классификация: Client-Side Monolith (SPA) с прямым доступом к BaaS.**

```
┌─────────────────────────────────┐
│   React SPA (Браузер клиента)   │
│  ┌───────────────────────────┐  │
│  │ Бизнес-логика             │  │
│  │ Финансовые расчёты        │  │
│  │ Валидация                 │  │
│  │ Управление состоянием     │  │
│  └───────────┬───────────────┘  │
└──────────────┼──────────────────┘
               │ Firestore SDK (прямое подключение)
               ▼
┌─────────────────────────────────┐
│  Firebase (Google Cloud)        │
│  ┌───────────────────────────┐  │
│  │ Firestore (NoSQL)         │  │
│  │ Auth (Google SSO)         │  │
│  │ Security Rules            │  │
│  └───────────────────────────┘  │
│  Cloud Functions: ❌ НЕТ       │
│  Pub/Sub: ❌ НЕТ              │
│  Cloud Tasks: ❌ НЕТ          │
└─────────────────────────────────┘
```

**Фатальный архитектурный выбор:** ВСЯ бизнес-логика выполняется в браузере. Нет серверной стороны. Это означает:

1. **Любой клиент может обойти валидацию** — открыв консоль браузера и вызвав Firestore SDK напрямую.
2. **Firestore Rules — единственный барьер** — и они проверяют только аутентификацию + роль (admin/не admin), но НЕ бизнес-правила (например, не проверяют что amountPaid ≤ totalAmount).
3. **Невозможно реализовать** сложную серверную логику: триггеры, очереди, расписание, webhook'и.

## 2.2 Качество доменных границ

| Граница | Оценка | Проблема |
|---------|--------|----------|
| Types отделены от логики | ✅ Хорошо | types/ директория |
| Services от hooks | ✅ Хорошо | services/ ← → hooks/ |
| Components от бизнес-логики | ⚠️ Средне | App.tsx = God Component (688 строк), вся оркестрация в одном месте |
| Финансы от операций | ❌ Плохо | calculateBaseTotals вызывается и для баланса, и для UI |
| Inventory от Sales | ⚠️ Смешано | salesAtomicService объединяет 5 коллекций в одну транзакцию — правильно для consistency, плохо для модульности |

## 2.3 Уровень связанности (Coupling)

**ВЫСОКИЙ.** `AppContent` компонент держит ВСЕ данные приложения в state:
- 14 хуков для данных (useProducts, useOrders, useTransactions, usePurchases, useClients, useEmployees, useFixedAssets, useWorkflowOrders, useJournal, useSettings, useBalance, useExpenses, useAppHandlers, useDebtRecalculation)
- Каждый хук = Firestore onSnapshot подписка
- Все данные передаются через props drilling (не через контексты)
- `Sales` компонент принимает **18 props**

## 2.4 Масштабируемость будущего расширения

**Очень ограничена:**
- Добавление нового модуля = изменение App.tsx + types.ts + новый service + новый hook + Firestore rules
- Нет plugin-архитектуры
- Нет event bus / message broker
- Нет feature flags инфраструктуры
- Lazy loading компонентов ✅ (React.lazy + Suspense)

## 2.5 Модель масштабирования

| Параметр | Статус | Детали |
|----------|--------|--------|
| Горизонтальное масштабирование | ❌ Не применимо | Нет backend серверов для масштабирования. Firebase масштабирует Firestore автоматически, но... |
| Stateless backend | ❌ Нет backend | Вся логика в браузере |
| Очереди | ❌ Нет | Нет асинхронной обработки |
| Фоновые задачи | ❌ Нет | Нет Cloud Functions/Cloud Tasks |
| Event-driven архитектура | ❌ Нет | Нет Pub/Sub, нет event sourcing |
| Кэширование | ⚠️ Только Firestore persistence | IndexedDB offline cache + localStorage для settings |
| Read replicas | ❌ Нет | Один Firestore instance |
| Изоляция отчётности | ❌ Нет | Отчёты формируются из тех же данных в реальном времени |

## 2.6 Data Layer

| Параметр | Оценка | Детали |
|----------|--------|--------|
| Transaction isolation | ⚠️ Serializable (Firestore транзакции) | Но только salesAtomicService использует полноценные транзакции |
| ACID compliance | ⚠️ Частичная | Атомарность только для продаж. Закупки, удаления — НЕ атомарны |
| Write contention | 🔴 Высокий риск | Firestore транзакции ретраятся при конфликте. При >10 конкурентных продажах одного товара — каскад ретраев |
| OLTP/Reporting separation | ❌ Нет | Один Firestore для всего |
| Multi-tenant | ❌ Нет | Единственный проект Firebase, одна коллекция для всех данных |
| Data leakage | 🔴 Риск | Любой аутентифицированный пользователь видит ВСЕ данные (Firestore rules: `isAuthenticated()` = read) |
| Migration discipline | ⚠️ Минимальная | `batchCreate` для первичной загрузки, но нет версионирования схемы |
| Backup strategy | ❌ Нет | Нет автобэкапов, нет точек восстановления, полагается на Firebase встроенный backup |

## 2.7 Архитектурный потолок

**~50 активных пользователей, ~100K документов в Firestore.**

После этого порога:
- Баланс `computeBalance()` загрузит ВСЕ документы в браузер → OOM
- Firestore `onSnapshot` для всех коллекций → bandwidth explosion
- Отсутствие пагинации → загрузка тысяч заказов при каждом открытии

# ФАЗА 3 — АУДИТ ФИНАНСОВОЙ ЦЕЛОСТНОСТИ (ERP-СПЕЦИФИКА)

## 3.1 Двойная запись (Double-Entry)

**ОТСУТСТВУЕТ ПОЛНОСТЬЮ.**

Система использует модель «мореходной книги» — каждая операция записывается один раз. Нет:
- Плана счетов (Chart of Accounts)
- Журнала проводок (General Ledger)
- Дебет/кредит записей
- Контроля баланса через T-счета
- Сверки (Reconciliation)

**Последствия:**
- Невозможно пройти аудит по МСФО/НСБУ
- Невозможно обнаружить ошибки через пробный баланс (Trial Balance)
- Ретроспективные изменения не отслеживаются через проводки

## 3.2 Надёжность расчёта баланса

**НЕНАДЁЖНА для enterprise-использования.**

`computeBalance()` в `balanceService.ts`:

```
Инвентарь = Σ(product.quantity × product.costPrice)  // по всем товарам
Выручка = Σ(order.subtotalAmount)  // по всем заказам
Себестоимость = Σ(item.quantity × item.costAtSale)  // по всем позициям заказов
Нераспределённая прибыль = totalAssets - [всё остальное]  // ЗАТЫЧКА
```

Проблемы:
1. **Нераспределённая прибыль = балансирующий элемент.** Если инвентарь некорректен, retained earnings автоматически компенсирует ошибку. Баланс ВСЕГДА сходится, даже если данные неверны.
2. **Equity = Σ purchases.amountPaidUSD** — это не собственный капитал, а сумма оплат поставщикам. Грубая ошибка в финансовой модели.
3. **Полный пересчёт при каждом изменении** — O(N) по ВСЕМ документам. Нет инкрементальных обновлений.

## 3.3 Риски консистентности PnL

| Риск | Серьёзность | Описание |
|------|-------------|----------|
| Двойной подсчёт | ⚠️ Средний | `calculateBaseTotals` имеет anti-double-counting логику, но она основана на хрупких эвристиках (проверка ID в множествах, проверка paymentMethod) |
| Несоответствие COGS и инвентаря | 🔴 Высокий | `costAtSale` фиксируется при продаже, но если costPrice товара меняется задним числом — COGS не пересчитывается |
| Расходы ↔ транзакции дублирование | ⚠️ Средний | Расходы могут создавать транзакции, поэтому нужна дедупликация по ID |

## 3.4 Race conditions в финансовых записях

### Критический: TOCTOU в `transactionService.update()`

```typescript
// ШАГ 1: Чтение ВНЕ транзакции (может устареть)
const oldSnap = await getDoc(docRef);  
const oldData = oldSnap.data();

// ШАГ 2: Транзакция (использует устаревшие данные)
await runTransaction(db, async (tx) => {
    // oldData уже может быть неактуальным!
    const oldDebtImpact = calculateDebtImpact(oldData);
    // ...
});
```

**Сценарий атаки:** Два кассира одновременно редактируют одну и ту же оплату. Первый сохраняет, второй использует старые данные для расчёта дельты долга → долг клиента искажён.

### Высокий: Неатомарная закупка

```
1. purchaseService.add(purchase)  → ✅ Сохранено
   --- СБОЙ СЕТИ ---
2. productService.update(product) → ❌ Не выполнено
Результат: Закупка есть, а товар на складе не появился
```

### Средний: Конкуренция на горячих продуктах

Firestore транзакции ретраятся при конфликте. Если 5 кассиров одновременно продают один товар через `salesAtomicService` — до 4 транзакций будут ретраиться, увеличивая latency.

## 3.5 Целостность аудит-лога

**ХОРОШАЯ для текущей архитектуры.**

- `journalEvents` — append-only коллекция (Firestore rules: `allow create`, `allow update, delete: if false`)
- Immutable записи: ни один пользователь не может изменить или удалить событие
- **НО:** журнал не покрывает все операции системы (например, изменение настроек, обновление прайса не логируются)
- **НО:** лог ведётся клиентской стороной — злонамеренный пользователь может просто не вызвать `addJournalEvent()`

## 3.6 Безопасность пересчёта (Recalculation Safety)

`useDebtRecalculation` — фоновый хук, пересчитывающий долги всех клиентов:
- Debounced (2 секунды)
- Использует hash данных для детекции реальных изменений
- Guard от конкурентных запусков через ref

**Проблемы:**
- Пересчёт запускается в браузере КАЖДОГО пользователя → множественные параллельные записи одних и тех же данных
- Нет серверного триггера — если браузер закрыть до завершения, данные останутся неконсистентными

## 3.7 Векторы мошенничества

| Вектор | Риск | Описание |
|--------|------|----------|
| Манипуляция ценами | 🔴 Критический | Сотрудник может изменить `priceAtSale` или `costAtSale` через консоль браузера — серверной валидации нет |
| Создание фиктивных возвратов | 🔴 Критический | `client_return` транзакция создаётся клиентским кодом, сервер не проверяет существование оригинального заказа |
| Обход кредитного лимита | 🔴 Критический | `creditLimit` проверяется только в UI — Firestore rules не валидируют бизнес-логику |
| Удаление заказов | ⚠️ Высокий | Admin может удалить заказ (`orderService.delete`), при этом инвентарь не восстанавливается, а транзакции остаются |
| Манипуляция курсом | ⚠️ Средний | `exchangeRate` устанавливается клиентом при каждой операции — можно подставить выгодный курс |

## 3.8 Контроль доступа к финансовым операциям

| Операция | Контроль | Оценка |
|----------|----------|--------|
| Создание заказа | `isAuthenticated()` | ❌ Любой аутентифицированный пользователь |
| Создание транзакции | `isAdmin()` | ✅ Только admin |
| Удаление заказа | `isAdmin()` | ✅ Только admin |
| Изменение настроек | `isAdmin()` | ✅ Только admin |
| Создание расхода | `isAuthenticated()` | ❌ Любой пользователь |
| Изменение товара | `isAdmin()` | ✅ Только admin |

**Проблема:** Роль `admin` жёстко закодирована для super-admin emails (`jasurmc@gmail.com`, `jassurgme@gmail.com`). Добавление новых администраторов требует изменения исходного кода.

## 3.9 Необратимые операции

| Операция | Необратима? | Риск |
|----------|-------------|------|
| Удаление заказа | ✅ Да | Нет soft delete, нет восстановления |
| Reset данных | ✅ Да | Удаление всех операционных данных навсегда |
| Удаление транзакции | ✅ Да | Account payable/receivable не восстанавливаются |
| Запись амортизации | ✅ Да | Нет механизма rollback |

**ВЫВОД ПО ФАЗЕ 3: Система НЕ МОЖЕТ безопасно управлять enterprise-бухгалтерией.** Отсутствие двойной записи, серверной валидации и защиты от мошенничества делает систему пригодной только как внутренний инструмент малого бизнеса с полным доверием к сотрудникам.

# ФАЗА 4 — МОДЕЛЬ УГРОЗ БЕЗОПАСНОСТИ И COMPLIANCE

## 4.1 Risk #1: Отсутствие тенантной изоляции

**Статус: ОДНОТЕНАНТНАЯ СИСТЕМА.**

Все данные хранятся в одном проекте Firebase, в одних коллекциях. Нет:
- Разделения по компаниям/организациям
- Префиксов в документах
- Отдельных Firestore database для тенантов
- Row-Level Security

**Сценарий угрозы:** При масштабировании на несколько компаний — ВСЕ компании видят данные друг друга. Утечка ВСЕЙ бизнес-информации.

## 4.2 Risk #2: Повышение привилегий (Privilege Escalation)

**Сценарий 1 — Через Firebase Console:**
```
1. Сотрудник с ролью 'sales' открывает DevTools
2. Получает Firebase config (публичный, в bundle)
3. Инициализирует Firebase SDK напрямую
4. Вызывает setDoc на коллекцию users, меняя свою роль на 'admin'
5. → Firestore rules: "Users can update their own profile, BUT cannot change role"
6. → НО: это правило сработает ✅ (diff check на 'role')
```

**Сценарий 2 — Обход UI-ограничений:**
```
1. Сотрудник 'sales' не видит модуль 'Баланс' в UI
2. Но Firestore rules позволяют READ балансовых данных всем аутентифицированным
3. → Запрос: getDocs(collection(db, 'balance')) → все данные доступны
```

**Вывод:** UI-ограничения (checkPermission) НЕ ЯВЛЯЮТСЯ барьером безопасности. Любой аутентифицированный пользователь может прочитать ВСЕ коллекции через прямой Firestore SDK.

## 4.3 Risk #3: Утечка финансовых данных

**Все данные доступны любому аутентифицированному пользователю:**
- Зарплаты сотрудников (employees.salary)
- Банковские реквизиты (clients.bankAccount, clients.inn)
- Себестоимость товаров (products.costPrice)
- Финансовые показатели (balance/current)

**Нет шифрования на уровне полей.** Firestore хранит всё в plaintext (Firebase обеспечивает encryption at rest, но нет field-level encryption).

## 4.4 Risk #4: Злоупотребление API

**Нет rate limiting.** Злоумышленная сторона может:
- Выполнить тысячи записей в Firestore за секунды
- Исчерпать бюджет Firebase (DDoS через billing)
- Залить коллекции мусорными данными

**Единственная защита:** Firestore rules + естественные лимиты Firebase (500 writes/sec per document).

## 4.5 Risk #5: Жизненный цикл токенов

- Firebase Auth токены автоматически обновляются (1 час жизни ID token)
- `browserLocalPersistence` — токен хранится в IndexedDB, переживает перезагрузку
- **Нет механизма принудительной инвалидации** — если сотрудник уволен, его сессия может продолжать работать до истечения refresh token
- **Нет IP-ограничений** — доступ из любой точки мира

## 4.6 Risk #6: Подделка аудит-лога

Хотя Firestore rules запрещают update/delete журнала, **создание событий доступно любому аутентифицированному:**
```
allow create: if isAuthenticated(); // Можно писать ложные аудит-записи
```

Злонамеренный сотрудник может внести фейковые события, дискредитируя аудит-лог.

## 4.7 Risk #7: Внутренние угрозы (Insider Threat)

| Угроза | Вероятность | Воздействие |
|--------|------------|-------------|
| Кассир манипулирует ценами в заказе | Высокая | Прямые финансовые потери |
| Менеджер создаёт фиктивного клиента с долгом | Средняя | Искажение дебиторки |

| Разработчик с доступом к Firebase Console | Высокая | Полный доступ ко ВСЕМ данным |

# ФАЗА 5 — МАСШТАБИРОВАНИЕ ДО ENTERPRISE-НАГРУЗКИ

## 5.1 Симуляция нагрузки

### 10K пользователей

| Параметр | Оценка | Проблема |
|----------|--------|----------|
| Firestore reads | ~100K reads/час | Каждый пользователь подписан на 8-10 коллекций через onSnapshot |
| Баланс pересчёт | ~50K операций сложения в браузере | Все документы в памяти каждого клиента |
| Memory (клиент) | ~200-500 MB | Все заказы, продукты, клиенты, транзакции в React state |
| Firestore cost | ~$500-1000/мес | Чтения + подписки |
| **Статус** | 🔴 **НЕРАБОТОСПОСОБНО** | onSnapshot для всех коллекций без пагинации → timeout |

### 100K пользователей

| Параметр | Оценка |
|----------|--------|
| Firestore reads | ~10M reads/час |
| Memory per client | Crash (OOM) |
| Firestore cost | ~$50,000/мес |
| **Статус** | 🔴 **НЕВОЗМОЖНО** |

### 1M пользователей

**НЕВОЗМОЖНО без полной реархитектуры.** Система спроектирована для <50 пользователей.

### Тяжёлая финансовая отчётность

**Каждый пользователь, открывший вкладку "Отчёты", вычисляет PnL/CashFlow из ВСЕХ данных в браузере.** При 10К заказов и 50К транзакций:
- `calculateBaseTotals`: O(orders + transactions + expenses) → ~60K итераций
- `computeBalance`: O(products + orders + expenses + fixedAssets + transactions + clients + purchases) → ~80K итераций
- Время вычисления: 2-5 секунд на мощном устройстве, 10-30 секунд на мобильном

### Закрытие месяца (Month-End Spike)

**Нет понятия "закрытие месяца".** Все пользователи одновременно создают расходы, оплаты, коррекции:
- Write contention на горячих документах (settings, balance/current)
- Debt recalculation запускается у КАЖДОГО онлайн-пользователя одновременно
- Множественные параллельные записи `updateClient()` → ретраи Firestore транзакций

### Массовые конкурентные записи

**Максимум 1 запись/сек на один документ Firestore.** При 10 кассирах, продающих один товар:
```
кассир_1: salesAtomicService.commitSale(product_A) → OK
кассир_2: salesAtomicService.commitSale(product_A) → RETRY (conflict)
кассир_3: salesAtomicService.commitSale(product_A) → RETRY, RETRY
кассир_4: salesAtomicService.commitSale(product_A) → RETRY, RETRY, RETRY → TIMEOUT
```

## 5.2 Точки разрушения

| Точка | Порог | Причина |
|-------|-------|---------|
| **Баланс в браузере** | ~5K заказов | OOM на мобильных устройствах |
| **onSnapshot подписки** | ~50 одновременных пользователей | Bandwidth saturation |
| **Write contention** | ~10 конкурентных записей/товар | Firestore transaction retry limit |
| **Firestore billing** | ~$5000/мес | 500K+ reads/час |
| **Balance pересчёт** | ~10K документов суммарно | Блокировка UI на мобильных |
| **Bundle size** | Текущий: ~800KB chunks | Рост компонентов → Code splitting limit |

## 5.3 Scaling Cost Explosion

```
10 пользователей:    ~$25/мес   (текущее состояние)
100 пользователей:   ~$200/мес  (терпимо)
1000 пользователей:  ~$5,000/мес (проблематично)
10000 пользователей: ~$50,000/мес (невозможно без реархитектуры)
```

**Причина:** каждый onSnapshot = все документы коллекции × количество подключённых клиентов.

# ФАЗА 7 — ИНВЕСТИЦИОННЫЙ АНАЛИЗ РИСКОВ

## Risk 1: Взрыв технического долга

**ПОЧЕМУ опасно:** Система построена на паттернах, которые невозможно масштабировать. `App.tsx` (688 строк) — God Component, все данные загружаются в память, нет серверной логики. Каждая новая фича увеличивает сложность экспоненциально.

**КОГДА критично:** При >20 одновременных пользователях или >5 новых модулях.

**КАК смягчить:** 
- Модульная автономия: каждый модуль = свои данные, свои подписки, свой контекст
- Введение backend (Cloud Functions минимум)
- Разделение App.tsx на feature-based routing

## Risk 2: Рост стоимости масштабирования

**ПОЧЕМУ опасно:** Firestore billing = reads × subscribers. Текущая архитектура (`onSnapshot` без пагинации для ВСЕХ коллекций) создаёт квадратичный рост затрат: `cost ∝ documents × users`.

**КОГДА критично:** При >200 активных пользователей → $3000+/мес на Firestore.

**КАК смягчить:**
- Пагинация и ленивая загрузка
- Server-side aggregation (Cloud Functions для отчётов)
- Кэширование агрегатов (не пересчитывать баланс на каждое изменение)

## Risk 3: Зависимость от одного разработчика

**ПОЧЕМУ опасно:** Код написан одним человеком (`jasurmc@gmail.com` = super admin = единственный коммитер). Нет:
- Code review процесса
- Документации архитектурных решений (хотя есть MD-файлы с TODO)
- Onboarding-гайда для новых разработчиков

**КОГДА критично:** Немедленно. Bus factor = 1.

**КАК смягчить:**
- Найм хотя бы одного full-stack разработчика
- Документация архитектуры и принятых решений
- ADR (Architecture Decision Records)
- Парное программирование

## Risk 4: Взрыв доменной сложности

**ПОЧЕМУ опасно:** Бухгалтерский учёт, налоги, нормативное соответствие — домена с ВЫСОКОЙ регуляторной сложностью. Система обрабатывает НДС (12%), курсовые разницы, кредиторку/дебиторку, но без двойной записи.

**КОГДА критично:** При первом aудите или требовании налоговой инспекции.

**КАК смягчить:**
- Внедрение двойной записи с планом счетов по НСБУ (Узбекистан)
- Консультация с профессиональным бухгалтером
- Автоматическая генерация проводок из операций

## Risk 5: Feature Creep

**ПОЧЕМУ опасно:** Система уже покрывает 14 модулей (dashboard, inventory, procurement, sales, workflow, reports, CRM, staff, payroll, fixed assets, balance, journal, price list, settings) при уровне зрелости MVP. Расширение без укрепления фундамента → хрупкость.

**КОГДА критично:** Любая попытка добавить HR, логистику, производство, e-commerce.

**КАК смягчить:**
- Feature freeze на новые модули
- Фокус на укреплении существующих
- Приоритизация стабильности над функциональностью

## Risk 6: Блокеры enterprise-продаж

**ПОЧЕМУ опасно:** Ни один enterprise-клиент не подпишет контракт без:
- SOC 2 / ISO 27001
- SLA (99.9%+)
- Multi-tenancy
- On-premise option
- API для интеграций
- Аудит-trail полного цикла

**КОГДА критично:** При любой попытке продажи B2B enterprise.

**КАК смягчить:** 12-18 месяцев enterprise-hardening (см. Фазу 8).

## Risk 7: Операционная надёжность

**ПОЧЕМУ опасно:** Нет мониторинга, нет алертинга, нет health checks. Если Firestore упадёт или data corrupted — узнаете от пользователей через часы/дни.

**КОГДА критично:** При >10 пользователях (текущее состояние).

**КАК смягчить:**
- Firebase Performance Monitoring (бесплатно)
- Sentry для ошибок фронтенда
- Cloud Monitoring алерты на Firestore billing
- Uptime check через Cloud Scheduler

## Risk 8: Юридическая/регуляторная экспозиция

**ПОЧЕМУ опасно:** Система хранит:
- ИНН, банковские реквизиты, номера телефонов (PII)
- Зарплаты сотрудников (Sensitive)
- Финансовую отчётность (Regulated)

Без шифрования полей, без policies, без DPA с Firebase.

**КОГДА критично:** При жалобе клиента или проверке регулятора.

**КАК смягчить:**
- DPA с Google Cloud
- Field-level encryption для PII
- Data retention policy
- Privacy policy для пользователей

## 🟡 INVEST WITH CONDITIONS — ИНВЕСТИРОВАТЬ ПРИ УСЛОВИЯХ

### Обоснование:

**Что есть (положительное):**

1. **Реальный доменный продукт.** Не pet project — система покрывает реальные бизнес-процессы металлоторговли в Узбекистане: склад, закуп с landed cost, продажи, CRM, баланс, PnL, НДС, ОС, зарплата. Видна глубокая экспертиза торгового домена.

2. **Качественный `salesAtomicService`.** Одна из лучших реализаций atomic sale для Firestore: single transaction на 5 коллекций, stock validation, duplicate detection, OCC. Это показывает, что разработчик ПОНИМАЕТ проблему и МОЖЕТ решать правильно — но не сделал это повсеместно.

3. **Технологический стек.** React 19, TypeScript, Vite, TailwindCSS, Firebase — современный, поддерживаемый стек. Lazy loading, code splitting, offline persistence — грамотные оптимизации.

4. **Существующие тесты.** ~150 unit-тестов для утилит и чистых функций. Качество тестов высокое (edge cases, builder patterns, assertions). CI/CD с type-check + tests в pipeline.

5. **Рыночная ниша.** ERP для металлоторговли в Средней Азии — узкий, но незанятый рынок с реальным спросом. Dual currency (USD/UZS) — правильная архитектурная decion для региона.

**Что критически отсутствует:**

1. **Нет backend.** Это дисквалифицирующий фактор для enterprise. ВСЯ бизнес-логика в браузере. Один из самых серьёзных архитектурных рисков.

2. **Нет двойной записи.** Без General Ledger это не ERP, а продвинутая таблица. Ни один профессиональный бухгалтер не примет эту систему.

3. **Масштабируемость ~50 пользователей.** All-data-in-memory + onSnapshot без пагинации = архитектурный потолок.

4. **Bus factor = 1.** Один разработчик = один truck accident = потеря продукта.
